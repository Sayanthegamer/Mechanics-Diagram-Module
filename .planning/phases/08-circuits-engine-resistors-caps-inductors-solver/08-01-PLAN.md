# Plan 8-1: Circuit Engine Porting (Core Classes, Stamps, and Math Utilities)

## <objective>
Port the foundational mathematical classes, matrix operations, and component-specific MNA (Modified Nodal Analysis) stamping equations from the `Sayanthegamer/circuitjs` scratch clone to the project's source tree under `src/lib/diagrams/circuit/`.

## <execution_context>
The project uses Vanilla TypeScript + Vite. All codebase utilities use 2-space indentation. We are porting code from a scratch repository clone (`scratch/circuitjs/circuit-sim/src/engine/`) to our library.

## <context>
- Original engine elements are located in `scratch/circuitjs/circuit-sim/src/engine/`.
- Element types, matrix mathematics, and serialization are required by the core `Circuit` solver class (which will be ported in Plan 2).
- React components and frontend-specific code from the source files must be omitted during the port.

## <threat_model>
- **Numeric Overflow & Singular Matrices**: Division-by-zero or NaN values generated by singular matrix loops.
  - *Mitigation*: Ensure `matrix.ts` includes pivot check validation (e.g. check if diagonal absolute values fall below `1e-12` during decomposition).
- **Untrusted Input via Deserialization**: Malicious or malformed JSON payload in `serialization.ts` leading to prototype pollution or crashing the main process.
  - *Mitigation*: Strictly validate types, shapes, and properties of elements during JSON parsed object mapping in `serialization.ts`.
</threat_model>

## <verification>
Run `npm run verify` (or compilation tests) to verify code compilation. All files must compile without errors under TypeScript's strict rules.

## <success_criteria>
- Core files (`types.ts`, `matrix.ts`, `serialization.ts`) and elements (`elements/`) exist in `src/lib/diagrams/circuit/`.
- Matrix equations compile and do not dynamically allocate new matrix arrays during iteration.
- Build successfully parses all imported files under Vite compilation.
